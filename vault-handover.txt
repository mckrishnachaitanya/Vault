CORE ARCHITECTURE
-----------------
- 6-digit PIN → PBKDF2 key → AES-256-GCM encrypts entry array → localStorage
- localStorage keys: vault_data, vault_pin, vault_theme, vault_bio,
  vault_bio_cred, vault_lockout, vault_fails, vault_local_updated,
  vault_cloud_sync, vault_cloud_uid, vault_cloud_email, vault_last_sync,
  vault_sync_pending
- IndexedDB "VaultDocsDB": object stores "meta" (key 'manifest' →
  encrypted manifest array) and "files" (key = doc id → encrypted
  {data: base64String}). See "R6a SUMMARY" below.
- state{} holds: entries, currentPIN, unlocked, unlockedViaBio,
  stepUpDone, pinBuffer, pinMode, pinTemp, editId, activeFilter,
  currentNav, detailSource, failCount, preLockScreen, formDraft,
  bioGen, bioPending, cloudUser, cloudSyncOn, syncStatus,
  docsStepUpDone, docs[] (in-memory decrypted manifest)
- Auto-lock: 2min idle. Lockout: 5 fails → timed delay.
- Single file always. No build tooling. No frameworks.

ENTRY SCHEMA
------------
All:      id, type, name, fav, created, updated, customFields[], sensitiveOverride
app:      url, username, password, notes
wifi:     ssid, password, securityType, notes
note:     note
card:     cardHolder, cardNumber(raw digits only), expiry(MM/YY), cvv, pin, notes
bank:     bankName, accountNo, ifsc, accountType, netBankingId, netBankingPw, notes
identity: docType(aadhaar/pan/passport/dl/voterid/other), docName,
          docNumber(raw digits for aadhaar), idName, dob(DD/MM/YYYY),
          expiry(DD/MM/YYYY), nationality, issuingState, notes

BIOMETRIC RULES
---------------
- WebAuthn platform only. Credential ID stored in vault_bio_cred.
- Fresh app open: PIN required once to seed currentPIN in memory.
  Bio button shows but does NOT auto-trigger on fresh open.
- After first PIN entry: bio auto-triggers on every subsequent
  auto-lock, via state.bioPending + visibilitychange/focus listeners
  (WebAuthn requires document focus, which isn't guaranteed at the
  moment lockVault fires on Android Chrome).
- unlockedViaBio=true → step-up PIN required for Card/Bank/Identity
  view, Add/Edit/Delete/Export actions, and restoring an in-progress
  Add/Edit form after re-auth.
- stepUpDone=true after first step-up (resets on lockVault). Manual
  PIN unlock ALSO sets stepUpDone=true (re-entering PIN proves PIN
  knowledge, satisfies step-up for that session).
- Settings security actions always prompt PIN directly (bypass
  stepUpDone). Step-up condition: state.unlockedViaBio (not
  isBioEnabled).

SCREEN RESTORE AFTER RE-AUTH
----------------------------
- lockVault() captures state.preLockScreen (form/detail/fav/settings)
  before locking. For form-screen, also captures state.formDraft via
  captureFormDraft() (formState, customFieldsData, editId, all
  #form-body field values).
- restoreAfterUnlock() runs after successful unlock instead of always
  going home: form-screen requires step-up first if unlockedViaBio
  (cancel → discard draft, go home), else restores via renderForm() +
  restoreFormDraft(). detail-screen re-opens via openDetail() (re-checks
  step-up itself). fav/settings just re-shown. Default: home.
- Cold start always goes home after unlock (preLockScreen never set).

SECURITY TIERS
--------------
- Sensitive: Card, Bank, Identity (require step-up when bio unlock).
- Non-sensitive: App, WiFi, Note.
- Per-entry override via sensitiveOverride field.

NUMBER FORMATTING
-----------------
- cardNumber and Aadhaar docNumber stored as raw digits (no spaces).
  Formatted on display only via formatCardNumber()/formatAadhaar():
  15 digits → Amex 4-6-5, 16 → standard 4-4-4-4, 12 (Aadhaar) → 4-4-4.
  Other lengths shown as-is. Card list subtitle reads raw digits.

RULED OUT — PERMANENTLY
-----------------------
- Password generator: not part of Vault scope, ever.
- Multi-page architecture: single file is intentional for security.
- Storing PIN or decryption key in Firestore: PIN never leaves device.

R5 SUMMARY (facts relevant to future sync/storage work)
---------------------------------------------------------
- Firebase loaded via CDN ES module dynamic import() (firebase-app,
  firebase-auth, firebase-firestore v10.12.2), lazy-loaded only when
  sync is touched — no build step. This is the pattern to follow for
  any future CDN library (e.g. JSZip for R6a).
- Auth: signInWithPopup, falls back to signInWithRedirect on
  popup-blocked/cancelled (this device always uses redirect — handled
  via vault_sync_pending flag + getRedirectResult() in restore()).
- Firestore doc: users/{uid}/vault/data — {data: <encrypted blob>,
  updated: <ms timestamp>}. Conflict resolution: last-write-wins by
  `updated`, whole-blob replace, no per-entry merge (by design).
- CRITICAL LESSON (caused real data loss, now fixed): timestamp-only
  "newer wins" is unsafe when one side can be "newer but empty" (e.g.
  a fresh PIN setup sets vault_local_updated to "now" with entries=[]).
  Fix pattern used everywhere blobs are compared: empty-local +
  non-empty-remote ALWAYS pulls down regardless of timestamps;
  non-empty-local + empty-remote ALWAYS pushes up; an incoming update
  that decrypts to empty while local is non-empty is IGNORED (logged).
  Apply the same emptiness-aware guard to any new sync/replace logic
  (R6a/R6b document blobs included).
- confirmDelete() previously had a fixed bug (undefined `overlay` —
  now resolved via document.getElementById('delete-overlay') at top
  of function). No other open bugs.
- Billing: Firebase project on Spark (free) plan, no payment method —
  quotas far exceed personal use, no surprise-bill risk.

R6a SUMMARY — implemented, not yet browser-tested
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- IndexedDB "VaultDocsDB" v1: stores "meta" (key 'manifest' → Crypto
  blob of [{id,name,mimeType,size,created,updated,fav,thumb}]) and
  "files" (key=id → Crypto blob of {data:base64String}). thumb is a
  data: URI (only for images), stored in the manifest for fast list
  thumbnails without decrypting the full file.
- DocsDB module: open/get/put/delete/loadManifest/saveManifest/
  loadFile/saveFile/deleteFile/reencryptAll(oldPin,newPin). Same
  Crypto.encrypt/decrypt as vault_data — no separate key derivation.
- state.docs holds the decrypted manifest in memory while unlocked;
  loaded via loadDocsManifest() on entering the Docs tab.
- Nav: Docs is the 5th bottom-nav slot (Home/Starred/+Add/Settings/
  Docs) on Home, Starred, Settings, and Docs screens. Lock moved to
  a top-right .lock-top-btn icon (id pattern lock-top-btn-{screen})
  on Home/Starred/Settings/Docs, same lockVault() handler.
- DOCS TAB GATE: openDocsTab() — if unlockedViaBio && !docsStepUpDone,
  calls _showStepUpModal() directly (bypasses requirePIN's
  stepUpDone shortcut). On success sets BOTH docsStepUpDone=true and
  stepUpDone=true (entering Docs satisfies step-up for the rest of
  the session, matching the "manual PIN unlock sets stepUpDone"
  precedent). docsStepUpDone resets in lockVault() alongside
  stepUpDone. No per-document sensitiveOverride — the tab is the gate.
- Add: file input (no capture attribute set — add later if camera
  capture for images is wanted) → fileToBase64() → DocsDB.saveFile()
  → unshift into state.docs → saveDocsManifest(). Inline upload row
  (.doc-upload-row, spinner) shown during encrypt+write.
- Viewer (docview-screen): images render via <img> from a blob: URL
  (base64ToBlob); PDFs via <embed type="application/pdf">; everything
  else shows a "no preview" placeholder. blob: URLs are revoked on
  back/delete/re-open to avoid leaks. Download/Share button uses
  navigator.share with a File if canShare({files}) is supported,
  else falls back to an <a download> click.
- Delete: same confirmDelete() sheet as entries, same step-up-if-bio
  gate as entry delete. Removes from manifest + DocsDB files store.
- PIN change (change-confirm): captures oldPin BEFORE reassigning
  state.currentPIN, then DocsDB.reencryptAll(oldPin, newPin) runs
  before the existing doExport() backup-export step.
- lockVault()/restoreAfterUnlock(): docs-screen and docview-screen
  (via currentDocDetail) added to preLockScreen capture/restore,
  same pattern as fav/settings/detail. Restoring into docview-screen
  re-enters via openDocsTab()+openDocDetail() so step-up re-fires
  correctly if needed.
- navigator.storage.persist() called once in init() (best-effort,
  no UI).
- EXPORT/IMPORT (separate from .vault entry backup): JSZip lazy-
  loaded from cdn.jsdelivr.net (same dynamic-import pattern as
  Firebase in R5). Export zip = manifest.json (PLAINTEXT metadata
  only — names/sizes/dates/fav, no file contents) + files/<id>.bin
  (the raw, still-encrypted Crypto blob from IndexedDB, written
  as-is — files are never decrypted to disk during export).
  Settings items: s-docs-export / s-docs-import, same
  step-up-if-bio gating as s-export.
- IMPORT PIN CAVEAT: imported file blobs were encrypted with whatever
  PIN was active at export time. Import only succeeds for files that
  decrypt under the CURRENT PIN (i.e. importing back into the same
  vault after a PIN change without re-export will silently skip
  those files — toast says "PIN mismatch?"). If R6b or a future
  session wants cross-PIN import, the zip would need to carry a
  PIN-independent wrapping key, which is a real design change — flag
  before building.
- id collisions on import (re-importing the same zip twice) are
  handled by generating a fresh id for any doc.id already present in
  state.docs.

WHAT WAS BUILT
--------------
- entry.linkedDocIds: string[] — preserved through form edits,
  absent on old entries (treated as []). Max 10 per entry.
- renderLinkedDocs(entry, container): appends a "Linked Documents"
  section to the detail-fields-container after entry fields.
  Shows each linked doc as a tappable row (icon + name + size),
  unlink ✕ per row, and "＋ Link a document" button.
- Link picker: bottom-sheet overlay (#link-picker-overlay) listing
  all docs not already linked to this entry. Tapping a row adds
  its id to entry.linkedDocIds and saves.
- Back-nav: state.docViewSource ('docs'|'entry') + state.docViewEntryId.
  When a linked doc is opened from an entry detail, back returns
  to that entry detail (not the Docs tab).
- Stale cleanup: cleanStaleLinkFor(docId) called on every doc
  delete (swipe and viewer). Scans all entries, removes the id,
  saves entries if changed.
- Step-up: "Link a document" button requires PIN if unlockedViaBio
  (same as edit button). Viewing linked docs does NOT re-prompt —
  the entry detail step-up already happened.
- docFileTypeMeta(doc): returns {icon, label, bg, color} for a
  doc — used in linked doc rows and picker. Parallel to existing
  DOC_FILE_TYPES/docFileMeta but with bg/color for icon backgrounds.

OVERVIEW
--------
Allow a vault entry (any type, but most useful for identity, bank,
card) to link to 1–10 documents from the Docs vault. Links show in
the entry detail screen; tapping a link opens the doc viewer.
Unlinking does not delete the document.

SCHEMA CHANGE
-------------
Add optional linkedDocIds: string[] to entry schema.
- Default: absent/undefined (treated as [])
- Max: 10 links per entry (enforced in UI — "Link another document"
  button hidden/disabled at 10)
- Existing entries with no linkedDocIds are unaffected — field is
  simply absent, no migration needed
- On doc delete: scan state.entries for any linkedDocIds containing
  that doc id, remove it from the array, call saveEntries(). This
  prevents stale links.

ENTRY DETAIL SCREEN CHANGES
----------------------------
New section rendered at the bottom of detail-fields, above the
edit/delete buttons (if any linked docs exist, or always as a
link-add affordance):

  LINKED DOCUMENTS
  [doc icon] HDFC_Statement_Jan.pdf    [✕]
  [doc icon] Cheque_Leaf.jpg           [✕]
  [+ Link a document]  (hidden when 10 links reached)

- Each linked doc row shows: docIcon(doc) + doc.name + unlink ✕
- Tapping the row (not ✕) opens the doc viewer for that doc
- After viewing, back button returns to entry detail (not docs tab)
  — detailSource logic needs extending: set a flag so docview-back
  goes to detail-screen instead of docs-screen
- Tapping ✕ unlinks (removes from linkedDocIds, saves entries,
  re-renders detail fields)

LINK PICKER
-----------
- "Link a document" opens a modal/sheet listing ALL docs (no
  category filter — user decides what to link)
- Docs already linked to this entry are excluded from the picker
  (greyed out or hidden)
- Picker shows same card style as docs list (icon, name, size, thumb)
- Tapping a doc in picker: adds its id to entry.linkedDocIds,
  saves entries, closes picker, re-renders linked docs section
- Picker is a simple overlay (same pattern as delete-overlay/
  stepup-overlay) — no new screen needed

BACK-NAVIGATION FROM LINKED DOC VIEW
-------------------------------------
When user opens a doc from an entry detail screen (not from Docs tab):
- Set state.docViewSource = 'entry' (vs 'docs' default)
- Also store state.docViewEntryId = entry.id
- In docview-back handler:
  if state.docViewSource === 'entry':
    showScreen('detail-screen'), re-open that entry detail
  else:
    openDocsTab() as before

STALE LINK CLEANUP
------------------
In docview-delete-btn handler, after deleting the doc from IndexedDB
and manifest, run:
  state.entries.forEach(e => {
    if(e.linkedDocIds && e.linkedDocIds.includes(id)){
      e.linkedDocIds = e.linkedDocIds.filter(lid => lid !== id);
    }
  });
  await saveEntries();
This keeps entries clean even if the detail screen is not currently open.

DISPLAY-ONLY RULES
------------------
- Linked docs section always visible in detail screen if entry has
  linkedDocIds.length > 0, even in bio-unlocked session (the doc
  viewer will enforce its own step-up if needed via openDocsTab
  which is NOT called in this path — doc is opened directly via
  openDocDetail(id), which has no step-up gate of its own).
  Decision: opening a linked doc from an entry detail does NOT
  require additional PIN beyond what was needed to open the entry
  detail itself. If entry was sensitive (card/bank/identity) the
  step-up already happened to open the detail. Consistent.
- "Link a document" button: requires step-up if unlockedViaBio
  (same as edit button on detail screen).

LIMITS
------
- Max 10 linked docs per entry. Enforced in picker and unlink
  logic. Toast: "Maximum 10 documents per entry" if attempted.
- No limit on how many entries can link to the same document
  (a single Aadhaar scan can be linked from multiple entries).
