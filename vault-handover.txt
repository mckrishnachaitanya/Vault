R7 SUMMARY — Multi-Entry Secure Share + ID Card Redesign (v6.0)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MULTI-SELECT (Home screen only)
- state.selectMode (bool), state.selectedIds (Set) — added to state{}.
  selectedIds intentionally persists across category-filter switches and
  search — this was a deliberate decision, not an oversight.
- ☑️ button in home-header-row (#select-mode-btn) toggles select mode.
  toggleSelectMode(on) — clears selectedIds when turning OFF.
- In select mode, entry-card shows a checkbox (.entry-checkbox) instead of
  fav-star/arrow; clicking the card toggles selection instead of opening
  detail. Swipe actions are disabled while in select mode (addSwipeActions
  is simply not called on that render pass).
- #select-bar (fixed above bottom-nav) appears once selectedIds.size > 0.
  Tapping the count OR the Share button opens the summary sheet — both do
  the same thing by design (summary-before-share was a late addition).
  ✕ button exits select mode entirely.
- openSummarySheet() — bottom sheet listing each selected entry with a
  remove (✕). Removing the last one auto-exits select mode.
  "Continue to Share" closes the sheet and calls openShareModal(entries[]).

SWIPE ACTIONS (Home + Starred entry lists only — NOT Docs)
- New addSwipeActions(container) — parallel implementation to the existing
  addSwipeToDelete(container) (which still powers the Docs list, untouched).
- Reveals 176px (two 88px buttons: Share, then Delete) instead of 88px.
  Layout order left→right = Share, Delete — Delete sits at the far edge by
  design (agreed as "furthest from content" during discussion, not a
  progressive-reveal threshold — both buttons appear together once the
  swipe crosses the open threshold).
- .swiped-actions class (176px margin) — distinct from the old .swiped
  (88px, still used by Docs) so both gestures coexist without CSS clashes.
- Swipe-share button calls openShareModal(entry) directly (single-entry
  quick share, same flow as the in-detail Share button).

SECURE SHARE — MULTI-ENTRY PAYLOAD
- openShareModal() now accepts a single entry OR an array — always
  normalizes internally to _shareTargetEntries[]. Detail-screen's Share
  button and the new swipe-share button pass a single entry (auto-wrapped).
- Share sheet title shows "🔗 Secure Share (N items)" when N > 1.
- Payload shape changed: { entries: [...], expiresAt }. OLDER LINKS with
  the legacy { entry, expiresAt } shape still decrypt fine — svAttemptUnlock
  checks payload.entries first, falls back to [payload.entry].
- Finishing a bulk share (Done button) auto-exits select mode.

ID CARD REDESIGN (Design D) — recipient view
- Old renderShareCard() + share-card-widget/.share-*-card gradient themes
  REMOVED entirely. Replaced by:
  - idCardModel(entry) — maps any entry type to {eyebrow, name, fields[]}.
    fields carry {label, value, emphasis, wide, prose} flags.
  - buildIdCardHTML(entry) — renders the actual .idcard markup: brass left
    edge, eyebrow row (type label + monogram), serif name, 2-col responsive
    field grid. Primary identifier (card/account/document number) gets
    emphasis+wide. Note/app "Notes" fields get prose (wraps, non-mono).
  - Cream/brass palette scoped under .idcard-scope (CSS vars prefixed
    --ic-*) so it never touches the app's own --accent/--bg vars.
  - Cormorant Garamond added to the Google Fonts link (display serif for
    the name); JetBrains Mono (already loaded) reused for field values —
    no extra font weight added.
- CVV/PIN/passwords are shown in full (NOT masked) — the whole point of
  the recipient view is revealing the actual secret. The masked look in
  the design mockups was a placeholder for the static preview only.

RECIPIENT CAROUSEL (multi-entry shares)
- #share-view-card now has fixed chrome (arrows, dots, peek edges, hint,
  count) around a single #share-idcard-slot that gets its innerHTML
  replaced per-slide — the old approach of overwriting the whole card
  container no longer applies.
- initShareCarousel(entries) — called once after successful PIN decrypt.
  Hides all carousel chrome when entries.length === 1 (single-entry shares
  look exactly like before, just with the new card design).
- renderShareCarouselSlide() — updates card content, dot states, arrow
  disabled states (first/last), and peek-edge visibility.
- shareCarouselGo(delta) — bounds-checked step; wired to both arrow clicks
  and a touchstart/touchend swipe listener on #share-carousel (40px
  horizontal threshold, must exceed vertical delta to count as a swipe).

R8 SUMMARY — Multi-Page Document Scan (v6.1)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

NEW DEPENDENCY
- jsPDF (UMD build) added via CDN in <head>:
  https://cdnjs.cloudflare.com/ajax/libs/jspdf/2.5.1/jspdf.umd.min.js
- Only used client-side to combine captured page images into one PDF —
  no server, no build step, consistent with the single-file architecture.

BEHAVIOR CHANGE (deliberate)
- EVERY camera scan now produces a .pdf, even a single-page scan.
  Previously single-page scans saved as .jpg. Unified to one code path
  (N pages → one PDF) since maintaining two save formats added complexity
  for no real benefit — a 1-page PDF opens identically to a jpg via the
  existing docview iframe preview.

CAMERA SCREEN — SESSION MODEL
- Camera stream now stays LIVE continuously through the whole scan
  session — it is only started once (openCameraScreen → startLiveView)
  and stopped only on camera-close or Done. Each capture just grabs a
  canvas frame from the still-playing <video>; this avoids the
  stop/restart flicker an earlier stop-per-page design would have had.
- State: _scanPages[] ({id, blob, url}), _scanSelectedId, 
  _scanRetakeTargetId. resetScanSession() revokes all object URLs and
  clears these — called on camera-close and on fresh openCameraScreen().

PAGE STRIP UI (#camera-page-strip)
- Appears once _scanPages.length > 0. Horizontal scroll of thumbnails,
  each with a page-number badge and a ✕ delete button.
- Tapping a thumbnail (not the ✕) toggles selection — selecting reveals
  #camera-page-action-row (Retake / Move ◀ / Move ▶ / Delete), matching
  the discussed "tap thumbnail, then tap arrows to move" pattern (NOT
  drag-and-drop — decided against it during discussion).
- Retake sets _scanRetakeTargetId and shows a toast; it does NOT reopen
  the camera (already live) — next capture-button tap replaces that
  page's blob+url instead of appending a new page.
- No page-count cap (explicit decision).

DONE → SAVE FLOW
- "Done (N)" button appears once ≥1 page captured, replacing the old
  single-shot retake button. Tapping it stops the stream, hides the
  strip/action-row/toolbar, shows the FIRST page as a static preview,
  and reveals the existing camera-confirm-bar (filename input unchanged
  from before, just no longer forces a .jpg extension).
- Save button: buildPdfFromPages() converts each page's blob → dataURL →
  Image → scaled to fit A4 (in points) preserving aspect ratio, one
  jsPDF page per image. Resulting blob wrapped in a File and passed to
  the EXISTING saveDocFromFile() — no changes needed there or in DocsDB;
  a multi-page scan is just a normal (larger) PDF file to the rest of
  the app.
- Button shows "Preparing PDF…" (multi-page) or "Saving…" (single) while
  the async PDF build runs; re-enables itself and restores its label if
  buildPdfFromPages() throws, with a toast — does not lose captured pages
  on failure, user can just retry Done → Save.

FALLBACK PATH (no getUserMedia)
- Unchanged from before: falls back to the native <input capture> camera
  via #doc-camera-input, which is single-shot only (no multi-page strip
  in this fallback — acceptable given it's already a degraded path for
  devices without getUserMedia support).

NOT DONE / WORTH KNOWING
- No per-page compression/resize beyond the existing JPEG 0.85 quality
  captured from canvas — multi-page PDFs from several full-res camera
  frames can still run several MB. Worth revisiting if this becomes an
  issue in practice (resize canvas before toBlob, e.g. cap long edge to
  ~2000px).
- No test device available this session to confirm real capture-button
  responsiveness/memory behavior with many pages (10+) in one session —
  logic is sound but worth a real test with a longer document.

R10 SUMMARY — Fixed Scan Save Hang (v6.3)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BUG REPORTED
- After capturing a page and tapping Save (PDF selected), the button got
  stuck on "Saving…" indefinitely — screen never advanced, no error shown.
  Screenshot confirmed camera-screen was still showing (i.e. execution
  never reached showScreen('docs-screen')), meaning the hang was inside
  buildPdfFromPages(), most likely jsPDF's CDN <script> tag not having
  finished loading yet (slow/first-time network fetch) when Save was
  tapped — the old code did `const { jsPDF } = window.jspdf` with no
  guard, so on a slow connection this could behave unpredictably instead
  of failing fast.

FIX
- waitForJsPDF(timeoutMs=6000) — polls for window.jspdf.jsPDF every 150ms
  instead of assuming it's already loaded. buildPdfFromPages() now awaits
  this before touching jsPDF at all.
- withTimeout(promise, ms, label) — generic timeout wrapper. The whole
  buildPdfFromPages() call is now wrapped with a 20s ceiling, so no
  matter what goes wrong inside (slow CDN, image decode issue, etc.) the
  Save button WILL recover instead of hanging forever.
- Graceful degradation on PDF failure:
  - Single page → falls back to saving the original JPEG automatically
    (the capture is never lost just because PDF generation failed), with
    a toast: "PDF unavailable — saved as JPEG instead".
  - Multi-page → can't meaningfully fall back (no single-file equivalent),
    so shows a clear error toast and RESTORES the button + keeps all
    captured pages so the user can just tap Save again without rescanning.

NOT YET DONE
- jsPDF is only requested for the first time when Save is tapped. Could
  optimize by triggering the script load earlier (e.g. the moment
  openCameraScreen() runs) so it's likely already warm by the time the
  user finishes scanning — not done this session, current fix guarantees
  correctness (no hang) but doesn't address perceived first-save latency
  on a slow connection.

R11 SUMMARY — jsPDF Load Retry Hardening (v6.4)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BUG REPORTED (with screenshot)
- Even with R10's timeout fix in place, Save still got stuck on
  "Preparing PDF…" — this time on a SINGLE page with JPEG explicitly
  selected (which per the code should never touch the PDF path at all),
  and the user reported multi-page scans failing right after too.
- The JPEG-selected-but-still-"Preparing PDF" detail is the important
  clue: that specific combination cannot happen from the app logic
  itself (wantsPdf is false whenever pages.length===1 && format==='jpeg'
  — confirmed by re-reading the exact shipped code). The far more likely
  explanation is the phone was still running a CACHED older copy of
  index.html via the service worker, from before the R10 fix — the
  vault-info.json version-check only shows an update NUDGE banner, it
  doesn't force the SW to serve fresh bytes immediately, so a stale
  cached script can keep running until the SW's own lifecycle rotates
  it out (typically requires fully closing and reopening the PWA,
  sometimes twice).
- I DO NOT have sw.js in this session's project files, so I could not
  inspect or fix the actual caching strategy. If this keeps recurring
  after confirming the version number in Settings → About matches the
  latest whatsNew entry, the fix likely belongs in sw.js's cache
  strategy for index.html (e.g. network-first instead of cache-first),
  not in index.html itself.

HARDENING ADDED REGARDLESS (belt-and-suspenders, doesn't rely on the
cache theory being correct)
- injectJsPDFScript() — if jsPDF isn't available after the first 5s
  poll, actively injects a FRESH <script> tag pointing at the same CDN
  URL and polls again (10s) before giving up. Previously the code only
  ever polled for the ORIGINAL <head> script tag to finish — if that
  particular request had failed outright (not just "still loading"),
  no amount of waiting would ever have helped; this retry gives it a
  second real attempt.
- openCameraScreen() now fires a fire-and-forget
  `waitForJsPDF(3000).catch(()=>{})` warm-up the moment the camera opens
  — by the time the user finishes capturing pages and taps Done→Save,
  jsPDF has had the whole scanning duration to load, not just the
  few seconds after Save is tapped.
- Overall PDF-build timeout raised 20s → 28s to give the new retry path
  room to actually complete rather than timing out mid-retry.
- Single-page JPEG fallback and multi-page "keep pages, show error"
  behavior from R10 are unchanged — this session only strengthens how
  hard the code tries before falling back / giving up.

R12 SUMMARY — Fast-Fail Multi-CDN jsPDF Loading (v6.5)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BUG REPORTED (confirmed on v6.3, ruling out caching as the cause)
- First scan of a session works; subsequent scans in the same session
  get stuck on the Saving/Preparing PDF button, requiring the camera's
  ✕ to escape. No stale-cache explanation this time — Settings → About
  confirmed the correct version was running.

ROOT-CAUSE REASONING
- Once jsPDF loads successfully, `window.jspdf` stays defined for the
  rest of the page's life — so if PDF generation had genuinely worked
  once, it should be INSTANT on every later attempt, not slower. The
  fact that failures kept recurring points to jsPDF never actually
  finishing its FIRST load reliably on this network — i.e. the earlier
  "successful" attempts were likely JPEG saves (no jsPDF involved at
  all), and every attempt that actually needed jsPDF for the first time
  has been hitting a slow/blocked cdnjs.cloudflare.com fetch. R10/R11's
  fixes prevented an infinite hang, but the worst-case wait was still
  up to 28 seconds with no visible progress — long enough that tapping
  ✕ before it resolved looked identical to a true hang.
- I attempted to fetch the actual jsPDF library file to embed it
  directly in index.html (removing the CDN dependency entirely), which
  would be the most bulletproof fix, but my tools could only retrieve
  it as opaque binary, not usable source text — so full inlining isn't
  possible from my end this session. Worth revisiting if this keeps
  recurring — bundling jsPDF's source directly into index.html (Krishna
  pasting the library code, or fetching it via a different method) would
  eliminate the network dependency for good.

FIX (this session)
- Two CDN mirrors now attempted in sequence, not just one:
  cdnjs.cloudflare.com and cdn.jsdelivr.net (JSPDF_SOURCES[]). If the
  first is slow/blocked, the second gets a real, fast, independent shot.
- Every stage is now individually time-boxed instead of one big
  best-effort wait: initial "is it already loaded" check (2.5s), each
  CDN script tag load (4s), each post-load poll (3s). Worst case across
  both mirrors is ~16.5s — down from 28s — and each failure surfaces
  quickly rather than sitting silently.
- Multi-page PDF failure no longer just shows an error and stops: the
  captured pages are now saved as SEPARATE numbered JPEGs
  (`name_page1.jpg`, `name_page2.jpg`, …) as a last-resort fallback, so
  a scan session's photos are never simply lost to a PDF-library
  failure. User sees "PDF unavailable — saved as N separate JPEGs
  instead".
- Single-page fallback behavior (auto-save as one JPEG) is unchanged
  from R10/R11.

NOT DONE / WORTH KNOWING
- jsPDF is still CDN-dependent, just more resilient now (2 sources,
  fast timeouts). If PDF generation keeps failing even after this,  the
  real fix is bundling the library's source directly into index.html so
  there's zero network dependency for this feature — flagged above,
  blocked on getting the actual library source text into this session.
- No live device available to verify actual timing/behavior on Krishna's
  real network — the timeout numbers are reasoned estimates, not
  measured against the actual failure conditions.

R13 SUMMARY — Real Root Cause Found: Save Button Never Reset After Success (v6.6)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BUG REPORTED (on confirmed v6.5)
- First scan of a session (3 pages → PDF): worked correctly.
- Second scan in the SAME session (1 page, JPEG default): Save appeared
  to do nothing — button showed "Preparing PDF…" even though JPEG was
  the visibly selected format.

ACTUAL ROOT CAUSE (this was NOT a jsPDF/CDN problem — R12's multi-CDN
work was solving a real but secondary issue, not this one)
- #camera-save-btn is a single reused DOM node across every camera
  session (screens are hidden/shown, never recreated). Its `disabled`
  flag and `textContent` were only ever restored in the FAILURE catch
  block — a SUCCESSFUL save left the button disabled with whatever
  label it had mid-operation ("Preparing PDF…" from the first, 3-page
  session) and never set it back to disabled=false / "Save to Vault".
- Next time the camera opened, the Save button inherited that stuck
  disabled+mislabeled state from the very first render. Because native
  `disabled` buttons don't fire click events at all, tapping it on the
  second attempt did literally nothing — matching "the app says saving
  pdf... the feature now not working" exactly. The visible JPEG
  selection was accurate; the button just couldn't respond to being
  tapped.

FIX
- openCameraScreen() now unconditionally resets #camera-save-btn to
  disabled=false / "Save to Vault" (and re-enables the JPEG/PDF toggle
  buttons) at the START of every session — regardless of how the
  previous session ended. This is the actual fix for the reported bug.
- Belt-and-suspenders additions:
  - Save handler now guards `if(btn.disabled) return;` at the very top,
    so no re-entrant double-invocation can ever occur even if something
    else fails to reset it.
  - The JPEG/PDF toggle buttons are now disabled for the duration of an
    in-flight save (re-enabled on failure or on the next session open),
    so switching format mid-save can no longer create a mismatch
    between the visible toggle and the operation already running.

RELATIONSHIP TO R12's CDN WORK
- The multi-CDN / fast-timeout jsPDF loading changes from last session
  are still valid and kept — they address a real (separate) risk of
  slow/blocked CDN fetches. But they were NOT the cause of this specific
  bug. This session's fix is the one that actually resolves the "second
  scan doesn't work" report.

R14 SUMMARY — Tier 1 Rectangular Crop (v6.7)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SCOPE (as discussed)
- Tier 1 only: axis-aligned rectangular crop, no perspective/corner
  correction. On-demand from the page strip (Option B) — capture flow
  itself is completely unchanged.
- New "✂ Crop" action added as the first button in the existing
  page-action-row (alongside Retake / Move / Delete), shown when a
  thumbnail is selected.

NEW SCREEN — #crop-overlay
- Full-screen overlay (z-index:200, sits above camera-screen). Shows the
  selected page's full image with a draggable rectangle over it.
- 4 corner handles (28px touch targets) — dragging tl/tr/bl/br adjusts
  that corner's two edges independently (rect always stays
  axis-aligned, never rotates/skews — this is the Tier 1 boundary).
- Dragging inside the rectangle (not on a handle) moves the whole
  selection; edges clamp to the image bounds and a MIN_CROP_PX (40px)
  minimum size.

  rect, repositioned on every drag frame via renderCropRect().
- "Reset" button in the top bar restores the default inset (6% margin
  on all sides) rather than full edge-to-edge, since a slight inset is
  a more useful starting point for a document photo.

COORDINATE HANDLING (the part most likely to need revisiting)
- All drag math happens in DISPLAYED pixel space (the image's actual
  on-screen size within crop-stage, measured via getBoundingClientRect()
  right after the image loads — one requestAnimationFrame delay to let
  layout settle first).
- On "Apply Crop", displayed-space rect coordinates are converted to the
  image's NATURAL pixel space via scaleX/scaleY
  (naturalWidth/displayedWidth, naturalHeight/displayedHeight) before
  drawing to an offscreen canvas — this is what keeps the crop accurate
  regardless of how large the phone's camera resolution is vs. how the
  image is actually rendered on screen.
- No orientation/EXIF handling beyond what the browser's <img> already
  does natively — not expected to be an issue since pages here come
  from getUserMedia canvas captures (no EXIF rotation metadata to begin
  with), but worth remembering if crop is ever extended to imported
  files from the Files picker.

INTEGRATION WITH EXISTING PAGE STATE
- Applying a crop REPLACES that page's blob+url in _scanPages in place
  (same pattern as Retake) — revokes the old object URL, creates a new
  one, re-renders the strip. Everything downstream (Done, format
  choice, PDF build, JPEG fallback) operates on whatever blob is
  currently in _scanPages, so cropping needed zero changes to the

- Crop can be applied to a page more than once (crops the

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SCOPE
- Full-app audit: performance, tab navigation, scrolling, bottom sheets,
  dead code. All findings implemented this session (full scope approved).

PERFORMANCE — THE BIG THREE
1. CRYPTO KEY CACHING (largest win)
   - Old behaviour: every Crypto.encrypt/encryptRaw generated a fresh
     random salt → fresh 100k-iteration PBKDF2 derivation (~100-300ms on
     a phone) for EVERY save, star toggle, doc open, manifest load, and
     sync decrypt.
   - New: Crypto.getKey(pin,salt) memoizes derived keys (Map keyed
     pin|saltHex, caches the promise so concurrent callers share one
     derivation; failed derivations evicted for retry). encrypt/encryptRaw
     now reuse ONE stable per-vault salt (localStorage 'vault_salt',
     created lazily) so their key is a cache hit too.
   - SECURITY REASONING (important context for future sessions): AES-GCM
     security rests on the unique random 96-bit IV per message, which is
     UNCHANGED. PBKDF2 salt only needs to be unique per PIN. Stored blob
     format is byte-identical (salt‖iv‖ct base64 / {salt,iv,ct} raw) —
     all pre-v6.8 blobs decrypt unchanged, no migration.
   - Share links pass {ephemeralSalt:true} → still one-off random salt,
     fully self-contained, never embed the vault salt.
   - Cache lifecycle: SURVIVES lock (deliberate — matches the existing
     policy of keeping state.currentPIN in RAM for biometric re-unlock).
     Cleared + vault_salt rotated on PIN change (twice: before saveEntries
     and again after reencryptAll, so old-PIN keys never linger).
   - Verified by standalone node test: roundtrip, legacy-blob decrypt,
     share-salt separation, raw path, timing (45ms cold → ~0.4ms cached
     on desktop; phone delta is much larger).
2. GLOBAL NON-PASSIVE TOUCHMOVE REMOVED
   - The crop feature had document.addEventListener('touchmove',
     cropPointerMove, {passive:false}) attached PERMANENTLY — forcing the
     browser to wait on JS before every scroll frame app-wide. Now
     attachCropDragListeners()/detachCropDragListeners() scope all four
     document-level drag listeners to crop-overlay open/close via new
     closeCropOverlay() (wired at all 5 open/close sites incl. camera ✕).
3. jsPDF <head> SCRIPT NOW `async`
   - Was render-blocking: on a slow/blocked cdnjs, the whole app's first
     paint waited on the CDN. Likely a real cause of slow cold starts on
     Krishna's network. Runtime loader (R10-R12) already handles
     late/failed loads, so async is safe.

R16 SUMMARY — Google Drive Backup (v6.9)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WHAT / WHY
- R6b cloud document storage finally shipped — as Google Drive appDataFolder
  SNAPSHOT BACKUP + explicit restore, NOT live sync. Firestore entries sync
  is UNTOUCHED and remains the live cross-device layer. Drive is the
  disaster-recovery layer for entries + documents together.
- Model ported from Paisa Trail's CloudBackup (Krishna shared its source
  this session), adapted: change-triggered instead of daily (vault data is
  mostly static, unlike daily-growing expenses — Krishna's call), encrypted
  payloads instead of plaintext, resumable upload instead of multipart.

ARCHITECTURE
- OAuth: GIS implicit token flow (accounts.google.com/gsi/client, lazy
  <script> load). Client ID const DRIVE_CLIENT_ID (created this session in
  the VAULT Firebase/Cloud project — consent screen already In production,
  drive.appdata is non-sensitive, no verification needed). Token in memory
  only. Silent re-issue (prompt:'') attempted first; interactive consent
  ONLY from user-gesture paths (connect / Back Up Now / Restore). Background
  auto-backup is silent-token-only — on failure it persists error + dirty
  state, never pops a login mid-session.
- TWO independent Drive files per backup cycle: vault-entries-<stamp>.json
  and vault-docs-<stamp>.json. Each uploads only when its side changes, so
  editing a password never re-uploads MBs of documents.
- Dirty flags: vault_drive_dirty_entries / _docs in localStorage, set by
  saveEntries() and saveDocsManifest() (the single choke points for both
  sides), 2.5s debounce → runBackup. Flags PERSIST across failures/offline/
  lock — checkDriveOnUnlock() retries them. "Backup on change" degrades to
  "backup at next opportunity", never silent loss.
- Version pruning: keep newest DRIVE_KEEP_VERSIONS(=3) per kind. NOTE:
  design-chat idea "manual supersedes latest auto" was deliberately DROPPED
  during implementation — keeping the pre-manual snapshot is strictly safer
  and the version cap bounds growth anyway (flagged in session summary).
- Empty-snapshot guard on BOTH kinds (Paisa Trail lesson): an empty side is
  skipped (flag cleared). Consequence: deleting the last doc does not empty
  the Drive docs backup — old versions linger until other changes push them
  out. Accepted trade for wipe-safety.
- Payloads are CIPHERTEXT ONLY. entries file: {app,kind,schemaVersion:1,
  exportedAt,count,data:<vault_data blob as-is>}. docs file: {…,manifest:
  <encrypted manifest blob as-is>,files:[{id,salt,iv,ct} base64]}. Built as
  Blob-part arrays (no giant contiguous string). bytesToBase64/base64ToBytes
  are NEW chunked helpers (32KB slices — String.fromCharCode spread
  overflows on MB arrays; verified 3MB round-trip in node).
- Upload: RESUMABLE (uploadType=resumable, POST session → PUT blob) because
  multipart caps ~5MB and docs backups exceed it.
- RESTORE (replace, never merge): walks versions newest-first per kind,
  validates decryptability with CURRENT PIN + non-emptiness, skips bad/
  undecryptable versions. All undecryptable → PIN_MISMATCH toast (backup
  needs the PIN active when it was made). Entries: state.entries → 
  saveEntries() (stamps vault_local_updated=now so Firestore push carries
  restored state UP, never pulled back over). _restoring flag suppresses
  markDirty during restore. Docs: encrypted manifest blob written AS-IS to
  meta store, file records replaced, local files not in backup deleted,
  state.docs + _docsManifestLoaded refreshed in place.
- PIN change: change-confirm branch marks BOTH kinds dirty after
  reencryptAll (old-PIN backups become undecryptable; restore validation
  walks past them until pruned out).
- Connect-on-empty / unlock-on-empty (entries AND docs both empty) → offers
  restore instead of backing up (never uploads empty over good). Unlock
  path uses silent token only.

UI (Settings → new "Google Drive Backup" section after Cloud Sync)
- s-drive-connect (🗄️, arrow hidden when connected; desc = connect hint /
  "Connected — backs up automatically when data changes")
- s-drive-backup-now (⬆️): confirm sheet with doc count + ~size estimate
  (entries blob length + Σdoc.size×1.37 b64 overhead) → full upload of both
  kinds, clears both dirty flags. Doubles as the tap-to-retry after errors
  (drive-backup-line shows "Last backup failed <when> — tap to retry" in
  amber, else "Last backup <formatSyncTime>").
- s-drive-restore (⬇️): step-up PIN if unlockedViaBio (mirrors cloud-delete
  precedent), lists newest of each kind with timestamps in the confirm
  sheet, "Restore & Replace".
- s-drive-disconnect (⛔ red): confirm sheet; revokes token, clears local
  Drive state. Backups REMAIN on Drive (deliberate).
- All confirms reuse confirmDelete() sheet. renderDriveUI() called from
  init(), settings nav tap, and every backup state change.

INTEGRATION POINTS (grep anchors)
- saveEntries → DriveBackup.markDirty('entries')
- saveDocsManifest → DriveBackup.markDirty('docs')  [covers every doc

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CONTEXT
- Resumes the parked design from R7: entries have had Secure Share for a
  while (self-contained encrypted link, no cloud needed); this session
  finally ships sharing for DOCUMENTS themselves, which can't fit in a URL
  the way small entry payloads can.

ARCHITECTURE DECISION (design chat before any code)
- Drive stores CIPHERTEXT ONLY, everywhere, no exceptions — this was the
  key fork of the whole design. Two options were discussed: (A) upload the
  decrypted file with the share PIN only protecting the link, or (B)
  upload the file already re-encrypted with the share PIN, so the raw
  Drive URL is useless without the PIN. B was chosen — it keeps Vault's
  "Drive never sees plaintext" rule intact everywhere, at the cost of
  share.html needing to do real decryption work instead of a plain
  redirect to Drive's own viewer.
- Multi-file record (not zip): each doc gets its own Drive file
  (drive.file scope, "anyone with link" reader), and Firestore holds one
  small record per share — an ENCRYPTED manifest (filenames, mimeTypes,
  sizes, per-file downloadAllowed flags) plus the list of Drive file ids,
  expiry, revoked flag, owner uid. Recipient fetches/decrypts files
  per-tap, not all upfront.
- Per-share "Allow download" toggle (not per-file in v1, though the
  manifest already carries a per-file `dl` flag so per-file control is a
  future-proofed, zero-migration addition later). View-only just hides the
  download button + disables right-click/drag on the preview — explicitly
  a deterrent, not a security guarantee (screenshots/devtools always win
  against a determined recipient; decryption happens in their browser by
  design).
- Expiry hard-capped at 24h for any share touching Drive/Firestore
  (regardless of what the UI preset shows) — entries-only shares keep
  their existing up-to-7-day range since those never touch the cloud.

NEW MODULE — DriveShare (parallel to DriveBackup, separate OAuth scope)
- drive.file scope (NOT drive.appdata — that's backup-only and can't
  produce shareable links). Own GIS token client, own in-memory token
  cache. Files live in a VISIBLE "Vault Shares" Drive folder (find-or-
  create, id cached in localStorage vault_share_folder_id, re-verified —
  not just trusted — on each use since the user could delete it).
- uploadShareFile(): same resumable-upload pattern as DriveBackup (PUT
  after a POST session, not multipart — consistent with the >5MB lesson
  from R16).
- makePublic(): sets anyone/reader permission after upload.
- newShareId(): random 16-byte hex, used as both the Firestore doc id and
  part of the Drive filename (vault-share-<id>-<n>.bin).

SENDER PIPELINE (generateDocShare → later generalized, see R19.4)
- Entry points: ☑️ multi-select on the Documents screen (new select bar,
  mirrors Home's), and a "🔗 Create Secure Link" button in the doc viewer
  for a single document. Both step up to PIN entry first if unlocked via
  biometrics (matches existing entry-share precedent).
- Per doc: DocsDB.loadFile (vault-PIN decrypt) → Crypto.encryptRawShare
  (NEW: re-encrypts with the 4-digit SHARE pin, ephemeral salt, self-
  contained salt‖iv‖ct Blob — distinct from the existing encryptRaw which
  uses the vault pin and the vault's stable salt) → upload → make public.
- Manifest: {v:1, files:[{fid,name,mimeType,size,dl}]}, itself encrypted
  with the share PIN (ephemeral salt) before being written into Firestore.
- Local ledger (vault_doc_shares, encrypted with the VAULT pin, NOT the
  share pin — this is Krishna's own record of what he's shared, needed for
  cleanup/Manage Shares in R19.3) records shareId + Drive file ids + expiry
  + names, appended on every successful share.
- On any failure mid-pipeline: already-uploaded Drive files for THIS
  attempt are rolled back (best-effort delete) — a half-published share
  must never linger.

NEW STANDALONE FILE — share.html (root of GitHub Pages, OUTSIDE the PWA)
- Zero shared code with index.html by design — its own tiny Crypto object
  (PBKDF2 deriveKey + AES-GCM decrypt, mirroring the app's primitives
  exactly) so this file has no build step and no dependency on the app
  bundle at all.
- Firestore doc read via plain REST (https://firestore.googleapis.com/v1/
  projects/<id>/databases/(default)/documents/shares/{shareId}?key=…) —
  no SDK needed for one anonymous doc read. Not-found / revoked / expired
  ALL produce the exact same generic "This link is no longer available"
  message — deliberately indistinguishable (Krishna's call) so a recipient
  can't learn anything from which failure mode they hit.
- Drive file bytes fetched via ?alt=media&key=… (anyone-with-link + API
  key, no OAuth needed for reading a public file).
- API key is a SEPARATE, RESTRICTED key (Share_Vault in Cloud Console) —
  NOT the Firebase config key. HTTP-referrer-locked to the GitHub Pages
  origin, API-restricted to Drive API + Firestore API only. Deliberately
  different from the app's own Firebase key since this one is embedded in
  a page anyone can view-source.
- PIN entry → decrypt manifest → file list → PER-FILE fetch+decrypt on tap
  (not all upfront) → inline preview: images directly, PDFs via the same
  pdf.js version as the main app (canvas render, no iframe). View-only
  files hide the download button, show a badge, disable right-click/drag —
  same deterrent-not-guarantee framing as the sender-side toggle.

sw.js CHANGE (required for share.html to work at all)
- Added an early-return bypass in the fetch handler: any request ending in
  /share.html skips the service worker's app-shell logic entirely. Without
  this, a navigate-mode request to share.html would have been answered
  with the CACHED index.html instead — share.html must always hit network
  fresh, and must never be treated as part of the installed PWA.
- CACHE_NAME bumped (vault-v1→v2) alongside this so the fix actually
  reaches devices instead of serving stale JS.

BUG FOUND BY KRISHNA #1 — link preview looks like the main app
- Cosmetic only (link itself worked correctly). share.html had a <title>
  but no Open Graph tags, so messaging-app preview cards showed generic
  branding indistinguishable from the main app's own link.
- Fix: og:title "🔐 Vault — Secure Document Share", og:description,
  noindex/nofollow, distinct favicon. Main app's <title> is bare "Vault"
  for contrast.

BUG FOUND BY KRISHNA #2 — download button inconsistent between two files
in the SAME share (real bug, not perception)
- Root cause: openViewer() is async (Drive fetch → decrypt → render) and
  reuses ONE dlBtn/body/title DOM set. Tapping file 2 before file 1's
  decrypt finished let both promises race to mutate the same elements —
  whichever finished LAST won, regardless of which file was on screen.
- Fix: monotonic _viewerToken counter. Every async checkpoint (fetch,
  decrypt, pdf.js load, each page render) re-checks its own token against
  the current one before touching the DOM; a superseded call silently
  no-ops instead of overwriting. Closing the viewer also bumps the token
  to kill any in-flight work for the file just closed.

R19.2 — Real root cause: idle auto-lock timer interrupting Drive OAuth (v7.4)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BUG REPORTED (multi-doc share via ☑️ select → Share)
- Tapping Encrypt & Upload triggered the Google account picker as
  expected, but completing it dropped the app to the PIN-lock screen
  (URL bar visibly changed to accounts.google.com and back — confirms a
  real top-level flow, not just a background tab). After re-unlocking and
  landing on Home, navigating back to Documents and starting a SECOND
  share attempt, the FIRST attempt's link appeared mid-way through
  entering the new share PIN — an old, abandoned request visibly hijacked
  the new one's screen.

FIRST FIX ATTEMPT (v7.3, INCOMPLETE — documented so it isn't re-tried)
- Added state.driveAuthPending counter, incremented around the interactive
  branch of DriveShare.getToken/DriveBackup.getToken, checked in the
  visibilitychange listener (mirroring the existing importInProgress
  pattern for the native file picker). Krishna re-tested: "no change".

ACTUAL ROOT CAUSE (found by asking Krishna to check Settings→About showed
v7.3 confirmed deployed, then whether the address bar genuinely navigated)
- Vault ALSO has a 2-minute idle auto-lock timer (setTimeout(lockVault,
  AUTO_LOCK_MS)) running completely independently of visibilitychange.
  Account-picker + consent screens can easily take longer than 2 minutes
  wall-clock with no touch/click on the original tab to reset it — so THIS
  timer fired lockVault() directly, bypassing the visibilitychange guard
  entirely since that guard was never consulted by the timeout path.

REAL FIX
- Moved the guard INTO lockVault() itself: `if(state.driveAuthPending>0)
  return;` as the very first line. This protects against every trigger
  path uniformly (idle timer, visibilitychange, anything added later) from
  one place, rather than needing every caller to remember to check.
- ALSO reordered generateDocShare (now generateHybridShare, see R19.4) to
  call `await DriveShare.getToken(true)` as the FIRST async operation,
  before ensureCloudUser()/Firebase SDK load/file decrypt — reduces the
  chance a mobile browser sees a stale user-gesture context by the time
  the popup is requested and silently converts it to a full top-level
  redirect in the first place.
- _shareGen monotonic generation counter added as a second safety net
  (independent of the lock bug): every UI-reveal point in the share
  pipeline re-checks it's still the current attempt before touching the
  screen. The share itself still completes and saves correctly either
  way — only the display of a superseded attempt is suppressed. (This is
  the same pattern as share.html's _viewerToken from R19.1.)

LESSON FOR FUTURE SESSIONS
- Any code path that can call lockVault() needs to be checked against
  BOTH the visibilitychange listener AND the idle timer — they are
  independent triggers and a fix to one silently doesn't cover the other.
  Prefer guarding inside lockVault() itself over guarding at each call
  site, as done here.

CLEANUP (cleanupExpiredDocShares, runs once per unlock — same setTimeout
batch as checkDriveOnUnlock etc.)
- Deleting the Firestore record IS the revoke — the read rule requires the
  record to exist and be unexpired, so removing it blocks recipient access
  immediately, same effect as it never existing.
- Drive file deletion happens AFTER the Firestore delete succeeds, and is
  best-effort/silent on failure — an orphaned ciphertext file with no
  surviving record pointing at it is harmless (nobody can rediscover its
  file id without the record that's now gone). If the Firestore delete
  itself fails (offline, etc.), the ledger entry is KEPT for retry next
  unlock — Drive is never touched until the record is confirmed gone.
- Ledger (vault_doc_shares, vault-PIN encrypted) is the source of truth for
  what to clean up; entries with exp>now stay untouched.

WHY THIS WAS NON-TRIVIAL (design chat before coding)
- Entry shares and doc shares were, until now, two architecturally
  different mechanisms: entries are fully self-contained (encrypted
  straight into the URL fragment, zero cloud dependency, up to 7-day
  expiry); docs require Drive+Firestore (24h cap). Attaching a document to
  an entry share necessarily means routing that share through the doc
  mechanism instead — there's no way to fit a document's bytes into a URL.
- Two decisions locked in with Krishna before writing code: (1) recipient
  gets ONE combined link/page (entry fields + documents together), not two
  separate links; (2) if Cloud Sync is off, the toggle is shown but
  disabled with "Enable Cloud Sync in Settings to include documents"
  rather than silently doing nothing.

SENDER SIDE
- Entries carry linkedDocIds (existing feature, up to 10 per entry,
  unrelated to sharing until now). collectLinkedDocs(entries) takes the
  UNION across all selected entries (bulk share), resolved to live doc
  objects (silently drops ids whose document was since deleted).
- Share sheet: new "Include documents" row, shown ONLY when the selected
  entry/entries actually have linked docs. Toggling it on reveals the same
  allow-download toggle + size/Wi-Fi-warning line already built for pure
  doc shares, and swaps the expiry presets to the 6h/24h set (7-day is
  hidden) exactly like doc mode.
- refreshShareSheetUI() replaced the old binary setShareSheetMode('entries'
  |'docs') — it's now a single function computing everything (include-row
  visibility/Cloud-Sync-gating, doc-opts visibility, expiry presets, button
  label) from current state, called after any state change (modal open,
  toggle click). This was a deliberate refactor, not just an addition —
  the old binary mode couldn't represent "entry mode WITH docs visible".
- generateDocShare was GENERALIZED into generateHybridShare(pin,
  {docsToUpload, entriesToInclude}) rather than duplicated — pure doc
  shares call it with entriesToInclude:null, entry+doc shares pass both.
  All the hard-won fixes from R19.1-R19.3 (gesture-first Drive auth,
  _shareGen guard, rollback-on-failure) apply to both cases automatically
  since there's only one implementation now.
- Manifest bumped to v2: adds an `entries` array (raw entry objects,
  same shape as the old pure-entry-share payload) alongside `files`.
  Backward compatible — a v1 manifest with no entries field at all is
  treated as an empty array by the recipient page.
- Plain entry shares (no docs, or toggle left off) are BYTE-IDENTICAL to

R20 SUMMARY — Secure Share full-scenario audit + hardening (v7.9)
------------------------------------------------------------------
Session trigger: Krishna felt "something wrong" with entry shares that have
linked documents, especially mixed multi-select (some entries with docs,
some without). Full code audit of both sides (index.html sender pipeline +

THE ROOT-CAUSE BUG (the one Krishna was feeling)
- openShareModal() called collectLinkedDocs() which reads state.docs — but
  the docs manifest is LAZILY loaded (loadDocsManifest) and reset on every
  lock (_docsManifestLoaded=false). Detail-screen Share was safe only
  because openDetail() loads the manifest; swipe-share and multi-select →
  summary → Continue called openShareModal() directly, so on a fresh unlock
  state.docs was empty and the "Include documents" row SILENTLY never
  appeared (or showed a wrong count). Session-dependent → felt intermittent.
- FIX: openShareModal is now async; awaits loadDocsManifest() first when
  any target entry has linkedDocIds. LESSON (generalizes): any NEW code
  path that reads state.docs must guarantee the manifest is loaded — grep
  for collectLinkedDocs/state.docs consumers when adding share entry points.

THE OTHER FIVE FIXES
1. Hidden-selected 6h expiry: selecting 6h in docs mode then toggling
   include-docs OFF hid the 6h button while it stayed "selected" —
   no visible preset, share silently got 6h. refreshShareSheetUI now
   resets to 24h in that direction too (mirror of the 7d→24h reset).
2. Custom fields: were in the encrypted payload but NEVER rendered by
   either idCardModel copy — recipients silently missed them. Now appended
   after the fixed per-type fields (wide rows, sender's order, sensitive →
   emphasis) in BOTH index.html and share.html. RULE: idCardModel exists
   in two files; any field-rendering change must be applied to both.
3. Bio step-up on entry shares: docs share had requirePIN when
   unlockedViaBio, entry shares did not — so "share entry + include docs"
   could decrypt+upload the very documents the Docs tab gates behind PIN.
   openShareModal now requires PIN step-up under bio unlock (no-op once
   stepUpDone). ALSO added to restoreAfterUnlock's share-sheet branch,
   because stepUpDone resets on lock — a bio re-unlock restoring the sheet
   would otherwise bypass the new gate.
4. Cancel-on-close + Copy Link: user-closing the sheet mid-hybrid-upload
   previously let the share complete invisibly — published, in the ledger,
   but the link never shown and Manage Shares had no way to retrieve it.
   Now: new _shareInFlight marker; ✕/overlay-tap during an upload bumps
   _shareGen ("Share cancelled" toast), and new gen checkpoints INSIDE the
   upload loop + immediately before the Firestore write make the existing
   catch/rollback delete anything uploaded. Auto-lock behaviour is
   DELIBERATELY unchanged: lockVault() hides the sheet without these
   handlers, so a backgrounded share still completes (R19.2 design).
   Manage Shares rows also gained a Copy Link button (rebuilds the URL
   from the ledger's shareId) — no link is ever unrecoverable now.
5. Recipient page (share.html): (a) exp was checked once at init — an
   already-open tab kept working past expiry; openViewer now re-checks exp
   on every file tap and flips to state-unavailable. Honest-client fix
   only — hard expiry still depends on sender's next-unlock Drive cleanup.
   (b) Blob URLs were never revoked (plaintext blobs held for page life);
   now tracked in _currentBlobUrl, revoked on viewer close and each open.

- Expiry is soft until sender's next unlock deletes Drive files (see 5a).

- cleanupExpiredDocShares silently skips if Firebase auth hasn't restored
  yet at unlock time; self-heals next unlock. Left as-is.

R22 SUMMARY — Revoke multi-prompt fix (v8.0.1)
-----------------------------------------------
Krishna reported revoking a share asked for the Google account multiple
times in ONE revoke. Root cause was threefold, all in DriveShare:
1. Revoke fired N un-awaited deleteFile() calls concurrently; with no
   cached token, ALL N started their own token request before any could
   populate the cache.
2. They shared ONE tokenClient whose .callback each call OVERWROTE —
   clobbering earlier callers' promises; GIS popped the picker repeatedly.
3. prompt:'consent' FORCED the full consent screen on every interactive
   request even when the scope was already granted.
FIXES: single-flight token acquisition (_tokenPromise — concurrent callers
JOIN the in-flight request; also collapses apiFetch's silent→interactive
fallback to at most one of each); prompt:'' so GIS shows consent only when
the grant doesn't exist; revoke's Drive deletes awaited SEQUENTIALLY (first
delete acquires the token on the tap's gesture, rest hit cache; still
best-effort per file — the Firestore delete IS the revoke). Side benefit:
rollback loop and cleanupExpiredDocShares use the same path, so their
popup-storm potential is gone too.
PATTERN (generalizes): any GIS token client shared across concurrent
callers MUST single-flight; per-call callback assignment on a shared
client is a landmine.

R23 SUMMARY — Unified Drive auth + recipient spacing (v8.0.2)
-------------------------------------------------------------
Krishna: app asks to authenticate at cloud-sync enable, at doc upload, AND
at revoke. Three surfaces: (1) Firebase Auth sign-in (one-time, persists —
fine); (2) DriveBackup GIS token, appdata scope; (3) DriveShare GIS token,
drive.file scope. (2)+(3) were SEPARATE token clients = separate grants =
separate prompts in the same session.
FIX: new DriveAuth object — ONE token client requesting BOTH scopes
(`${DRIVE_SCOPE} ${DRIVE_SHARE_SCOPE}`), holding the R22 single-flight.
DriveBackup.getToken and DriveShare.getToken are one-line delegations, so
every caller is unchanged. One prompt per session now covers backup,
share upload, and revoke. DriveBackup.disconnect() calls DriveAuth.reset()
— DELIBERATE: disconnecting Drive revokes the share scope too.
- MIGRATION NOTE: existing installs see ONE more consent screen after this
  update (approving the combined grant), silent thereafter. Verified on
  device.
- IRREDUCIBLE RESIDUE (not bugs): one prompt per cold start of the app,
  and re-prompt after ~1h token expiry — inherent to client-only implicit
  flow; GIS access tokens are memory-only and there's no server to hold
  refresh tokens. Do not attempt to persist bearer tokens to storage.
- share.html spacing bug (screenshot-reported): a group's doc rows sat
  nearly flush against the NEXT entry's card and read as belonging to it.
  Fix: in-group gap 6px (docs hug their own card), 26px margin between
  .entry-group blocks.
